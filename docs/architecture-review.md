# Architecture review

_Date: 2026-04-26_

A maintainability review of the pgForEachDb solution. Items are listed by priority. Each entry has a one-sentence reason; pick them up in any order.

## Priority 1 — Worth doing soon

### 1.1 Floating package versions — track the latest
All `PackageReference` entries float on `Version="*"` (deliberate — accept latest stable on every restore, including major bumps). Documented as a project policy.

**Reason:** Trades reproducibility for being current. Build can break on a transitive bump made hours ago, so:
- A `dotnet restore --force-evaluate` step in CI surfaces drift early.
- Optionally enable `<RestorePackagesWithLockFile>true</RestorePackagesWithLockFile>` and commit `packages.lock.json` if reproducible CI matters; the lockfile is regenerated by `dotnet restore --force-evaluate` whenever you want to pull the latest.

### ~~1.2 Replace or retire `tonerdo/readline`~~ — **done**
Replaced with `AnsiConsole.Prompt(new TextPrompt<string>("Password:").Secret())` in [`ForEachDb/Program.cs`](../src/ForEachDb/Program.cs). `ReadLine` package removed. Spectre.Console was already a dependency.


`ReadLine 2.0.1` (used in [`ForEachDb`](../src/ForEachDb/) legacy CLI) — last push **2021-08-19**, MIT license, technically not archived but functionally abandoned (28 open issues, no commits in ~5 years).

**Reason:** Unmaintained dependency in a security-relevant input path. Now that the desktop app exists, options are: (a) delete the legacy CLI, (b) replace `ReadLine` with `Spectre.Console`'s built-in `AnsiConsole.Prompt<T>` or `System.Console.ReadLine` (since the new TUI was already replaced with Spectre per commit `960f77e`).

### ~~1.3 Move log export into the domain library~~ — **done**
New [`LogExporter`](../src/ForEachDbQueries/LogExporter.cs) writes a tab-separated stream from `IEnumerable<DatabaseLogEntry>`. The VM keeps a parallel `_rawLogs` list and feeds the exporter in `ExportLogAsync`. Symmetric with `CsvExporter` now.


`WorkspaceViewModel.ExportLogAsync` (~lines 304-327) writes a tab-separated file inline (`StringBuilder` + `File.WriteAllTextAsync`). The parallel concept — CSV export — lives in [`CsvExporter`](../src/ForEachDbQueries/CsvExporter.cs) under the domain library.

**Reason:** Inconsistency. Symmetry between CSV and log export is one new file (`LogExporter.WriteAsync(stream, logs)`) that the VM can call. Also enables a cheap unit test instead of UI-only verification.

### ~~1.4 Extract a small filter helper for Results + Logs~~ — **done**
New [`FilterSelector`](../src/ForEachDb.Desktop/ViewModels/FilterSelector.cs) (`Values`, `Selected`, `Matches`, `RegisterValue`, `Reset`). The VM now exposes `ResultsFilter` and `LogsFilter` and subscribes to each one's `PropertyChanged` to trigger `ApplyResultFilter` / `RefreshLogVisibility`. XAML binds `{Binding ResultsFilter.Values}` / `{Binding ResultsFilter.Selected}` (and the log equivalents). Removes ~30 LOC of duplication and the loose `IsAllDatabasesFilter` helper.


[`WorkspaceViewModel`](../src/ForEachDb.Desktop/ViewModels/WorkspaceViewModel.cs) carries two near-identical filter flows:
- `DatabaseFilter` / `DatabaseFilters` (results pane)
- `LogDatabaseFilter` / `LogDatabaseFilters` (log pane)

Both use the same `(all databases)` sentinel, both rebuild on filter change, both clear on run start. ~40 lines of structural duplication.

**Reason:** Each subsequent filtered view (e.g. selectable run statuses, log levels) will multiply the pattern. A single `FilteredView<T>` (`Items`, `Filters`, `SelectedFilter`, predicate delegate) covers both today and is reusable for future panes.

## Priority 2 — Worth doing eventually

### 2.1 Keep an eye on `CommandLineParser`
Used in legacy CLI. Last push **2024-02-29**, MIT, not archived but slow. Modern alternatives: `System.CommandLine` (Microsoft, stable now), `Spectre.Console.Cli` (already a project dep).

**Reason:** Not urgent — it works. But if the CLI is kept, prefer consolidating on Spectre.Console.Cli at the next CLI change to drop one dependency.

### 2.2 `WorkspaceViewModel` is approaching the size threshold — **partially done**
Recipe save/load extracted to [`WorkspaceViewModel.Recipes.cs`](../src/ForEachDb.Desktop/ViewModels/WorkspaceViewModel.Recipes.cs) (partial class). Filter logic stays in the main file since it's already cohesive after 1.4. Main file now ~365 LOC.


~454 LOC, ~70 members. Recipe save/load (4 methods, ~40 lines) and the new log filter could move to small partial classes (`WorkspaceViewModel.Recipes.cs`, `WorkspaceViewModel.Filters.cs`) without changing behaviour.

**Reason:** Pre-emptive split before the file becomes hard to navigate. Partial classes preserve the `[ObservableProperty]` / `[RelayCommand]` source-generator wiring with no logic change.

### ~~2.3 Desktop app has zero tests~~ — **done**
New [`ForEachDb.Desktop.Tests`](../src/ForEachDb.Desktop.Tests/) project (NUnit + AwesomeAssertions, no Avalonia / UI-thread dependency). 40 tests cover:

- [`DatabaseVisibilityTests`](../src/ForEachDb.Desktop.Tests/DatabaseVisibilityTests.cs) — glob translation, substring fallback, template-hidden-by-default, plus an explicit regression case for the original `b*` regex bug.
- [`FilterSelectorTests`](../src/ForEachDb.Desktop.Tests/FilterSelectorTests.cs) — sentinel state, `RegisterValue` de-dup, `Matches`, `Reset`, `PropertyChanged` on `Selected`.
- [`LogRowTests`](../src/ForEachDb.Desktop.Tests/LogRowTests.cs) — level → flag mapping, level-string upper-casing, default visibility.
- [`ResultRowTests`](../src/ForEachDb.Desktop.Tests/ResultRowTests.cs) — cell/column alignment, null-to-empty, empty input.
- [`DatabaseItemTests`](../src/ForEachDb.Desktop.Tests/DatabaseItemTests.cs) — selection callback, badge/terminal-flag per `RunState`, `Reset`.

To make `RefreshVisibility` testable without spinning up the VM, the predicate moved into a new pure-function helper [`DatabaseVisibility.Matches`](../src/ForEachDb.Desktop/ViewModels/DatabaseVisibility.cs).

Suite runs in ~90 ms — fast enough to run on every save.

### 2.4 Two database-discovery paths
- [`DatabaseFinder`](../src/ForEachDbQueries/DatabaseFinder.cs) (server-side filter via SqlBuilder, used by the legacy CLI).
- [`DatabaseProbe`](../src/ForEachDb.Desktop/Services/DatabaseProbe.cs) (hard-coded SQL, used by the desktop app).

The desktop app deliberately bypassed `DatabaseFinder` because it now wants templates *and* a `datistemplate` flag in the result. There's no consumer overlap today.

**Reason:** Not a duplication problem yet, but a fragmentation risk. If the legacy CLI is retired (1.2), `DatabaseFinder` and most of `Dapper.SqlBuilder` go with it. If it stays, document why two paths exist.

## Priority 3 — Nice to have

Now catches `JsonException`, `IOException`, and `UnauthorizedAccessException`; corrupt or unreadable file is treated as empty.


IO errors (permission denied, file disappeared mid-read) propagate. The desktop calling site already wraps it, but the CLI path may not.

**Reason:** Minor robustness gap. Either widen to `Exception`, or document that the caller is responsible.

### ~~3.2 Avalonia `defaultBindingMode: BindingMode.TwoWay` on `BoundTextProperty`~~ — **done**
Removed; default is now Avalonia's standard one-way. The XAML site already specifies `Mode=TwoWay`.


Works today but is unusual for an attached property. A consumer that explicitly binds `OneWay` will be surprised.

**Reason:** Minor surprise factor; consider keeping `OneWay` default and requiring the binding to specify `Mode=TwoWay` explicitly.

## What is **not** worth doing

A few items considered and discarded — listed so they don't get re-raised.

- **A DI container** (Microsoft.Extensions.DependencyInjection or similar). The manual wiring in `MainWindow` → `ShellViewModel` is two lines and clear. A container would buy nothing here.
- **Splitting `WorkspaceViewModel` into per-pane sub-VMs** (`QueryEditorVM`, `ResultsVM`, `LogVM`). The panes share enough state — selection, run, filter — that this would add wiring without removing complexity.
- **An MVVM framework swap** (Prism, ReactiveUI). CommunityToolkit.Mvvm is the lightweight option and is enough.
- **Removing `Dapper.SqlBuilder`**. It's stable, small, and the legacy CLI uses it. Only revisit if 1.2 retires the CLI.
- **Three-layer conversion of results** (`DatabaseRow` → `AggregatedResults` → `ResultRow`). Each layer has a distinct purpose (domain shape → grid shape → cell binding). Not redundancy.
